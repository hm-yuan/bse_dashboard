# UI_STYLE_GUIDE.md

## 1. 设计定位

本项目是“交易所画像 Dashboard”，用于对外展示和内部演示。

第一版目标不是建设完整生产级业务系统，而是形成一个具备以下特征的现代化数据产品原型：

* 现代化
* 简洁
* 高级
* 专业
* 克制
* 可信赖
* 适合对外讲述
* 适合监管和资本市场场景

整体设计方向为：

> Fluent Light + Carbon Edge + BSE Brand。

本项目不直接照搬任何一个设计系统，而是形成适合交易所画像场景的自有风格：

> Fluent Light + Carbon Edge + 北京证券交易所品牌识别的现代金融监管 dashboard 风格。

该方向强调浅色商务空间、北京证券交易所 LOGO 品牌识别、清晰结构线、硬朗卡片、克制阴影和更强的信息对比。Fluent Light 负责浅色空间、现代感、简洁与克制；Carbon Edge 负责硬朗线条、明确边界和专业数据产品结构感；BSE Brand 负责使用北京证券交易所 LOGO 的蓝、红、青作为品牌识别来源；Ant Design Pro 负责中后台页面组织和信息密度控制。

## 2. 设计系统取舍

### 2.1 Fluent Light：主视觉方向

本项目整体视觉气质以 Fluent Light 为主要参考，但不采用过度柔和、过度圆润的表现。

借鉴内容：

* 现代卡片层级
* 轻量阴影
* 清爽的浅色空间
* 克制但有科技感的蓝色体系
* 适合展示型 dashboard 的空间感
* 统一的 design token 思路

使用场景：

* 页面整体布局
* KPI 卡片
* 信息卡片
* 洞察卡片
* 页面背景
* 按钮、筛选器、导航
* 轻量交互状态

设计目标：

> 让 dashboard 看起来现代、干净、高级，并保持金融商务与监管级数据产品的克制气质。

---

### 2.2 IBM Carbon Edge：结构线和数据表达规范

IBM Carbon Edge 是本项目结构感和监管级数据表达的重要参考。

借鉴内容：

* 克制的企业级数据产品风格
* 清晰的信息层级
* 硬朗的边框、分割线和标题区
* 卡片内容区与标题区的明确分隔
* 图表服务于数据故事
* 坐标轴、图例、标注的规范化
* 数据可视化中的可读性和一致性
* 风险色、状态色、对比色的克制使用

使用场景：

* 气泡图
* 热力图
* 四象限图
* 趋势图
* 结构矩阵
* 表格中的风险标签
* 图表标题和口径说明
* KPI 卡片、主图卡片、洞察区和明细表的结构线

设计目标：

> 让页面具备监管级、专业级的数据表达能力，而不是只追求柔和视觉效果。

---

### 2.3 Ant Design Pro：页面组织和中后台结构

Ant Design Pro 不作为主视觉风格，但作为页面结构和业务组织方式的重要参考。

借鉴内容：

* KPI 区
* 卡片式布局
* 详情表格
* 筛选区
* 导航结构
* Dashboard 页面分区
* 中后台信息密度控制

使用场景：

* 页面模块划分
* 明细表
* 筛选器
* 顶部指标区
* 首页总览
* 后续权限、菜单、详情页扩展

设计目标：

> 让页面结构清晰、可扩展、适合后续演变成业务平台，但避免出现明显模板感。

---

### 2.4 BSE Brand：北京证券交易所品牌识别

本项目导航和强调色应使用北京证券交易所 LOGO 的蓝、青、红作为品牌识别来源。

使用规则：

* 顶部导航栏左侧使用官方 LOGO 图片，不再使用单独的 BSE 蓝色胶囊标识。
* LOGO 必须保持原始比例，不得拉伸、变形、裁切。
* 主色使用 BSE 蓝。
* 青色用于辅助强调、KPI 顶部线、活跃度、流动性等正向辅助信息。
* 红色主要用于风险、异常和少量品牌点缀，不得大面积使用。
* 页面整体仍以白色、浅灰蓝、深蓝文字为主。

---

### 2.5 Atlassian：仅少量借鉴状态和流程表达

Atlassian 不作为本项目主视觉风格。

可借鉴内容：

* 状态标签
* 事项卡片
* 任务进度
* 处置状态
* 风险跟踪
* 流程节点

使用场景：

* 监管处置进展
* 风险状态标签
* 募投项目状态
* 上市审核流程
* 问询、监管措施、整改进度

设计目标：

> 少量吸收 Atlassian 在协作和状态管理上的优点，但不让整体界面变成项目管理或工单系统风格。

## 3. 总体设计原则

### 3.1 美学统一，结构差异化

页面之间不应千篇一律，但必须共享同一套视觉语言。

统一内容包括：

* 色彩体系
* 字体层级
* 卡片样式
* 图表语法
* 表格样式
* 图标风格
* 间距系统
* 阴影规则
* 圆角规则
* 风险色使用规则

差异化内容包括：

* 页面布局
* 主图位置
* 讲述节奏
* 辅助模块组合
* 页面叙事方式

### 3.2 每页只保留 1-2 张主图

本项目不是指标堆砌型大屏。

每个画像页面应采用：

* 1 句核心判断
* 5-6 个 KPI 卡片
* 1-2 张主图
* 3-5 条关键洞察
* 1 个明细钻取区

页面主图必须回答该页面的核心问题。

如果某个图表不能支撑一个明确判断，不应放在主页面，可放入明细区或后续二级页面。

### 3.3 先讲判断，再给证据

每个页面的阅读顺序应是：

> 核心判断 → KPI 状态 → 主图证据 → 关键洞察 → 明细钻取

不要让用户先看大量图表后再自己得出结论。

### 3.4 克制使用装饰

禁止使用过多大屏式炫光、粒子、强渐变、复杂背景、过度动效。
禁止使用过度毛玻璃、过度圆角、过度发光和大面积装饰渐变。

本项目应体现：

* 专业
* 稳重
* 清晰
* 精准
* 有设计感

而不是“炫酷大屏”。

### 3.5 Fluent Light + Carbon Edge + BSE Brand 视觉规则

1. 页面整体应浅色、硬朗、金融、克制，避免柔和过度。
2. 导航栏采用白色或近白色商务导航，不使用深色导航作为默认导航。
3. 导航栏左侧使用北京证券交易所 LOGO 图片，不再使用单独的 BSE 蓝色胶囊标识。
4. 当前页面激活态不使用圆润气泡、透明气泡、半透明块或毛玻璃背景。
5. 激活态优先使用字体加粗、深蓝或近黑色文字、底部 2-3px BSE 蓝色强调线。
6. 全局卡片圆角建议为 10-12px。
7. 卡片必须有明确边框、标题区分割线和克制阴影。
8. 页面背景应为克制浅灰蓝，不使用明显光晕、粒子或大屏式炫光。
9. 结构线用于建立层级，不用于装饰性堆叠。
10. 视觉重点来自结构、线条、排版和对比，而不是渐变、炫光或厚重色块。
11. LOGO 中的红色只作为少量品牌点缀或风险色，不应大面积使用。
12. LOGO 中的青色可作为辅助强调色，用于 KPI 顶部线、活跃度、流动性等正向辅助信息。

## 4. 色彩系统

### 4.1 主色

| 颜色          | 色值        | 用途             |
| ----------- | --------- | -------------- |
| 深蓝          | `#0B2A5B` | 页面标题、导航、核心文本   |
| BSE 蓝        | `#005BAC` | 当前页面、高亮对象、品牌强调  |
| BSE 深蓝      | `#003B7A` | 强调线、重点数据、激活状态 |

### 4.2 辅助色

| 颜色 | 色值        | 用途             |
| -- | --------- | -------------- |
| BSE 青 | `#00A6C8` | 交易活跃度、流动性、市场活跃 |
| 绿色 | `#22A06B` | 正向变化、改善、达标     |
| 橙色 | `#F59E0B` | 中度风险、提示、关注     |
| BSE 红 | `#E3232E` | 高风险、异常、预警、少量品牌点缀      |
| 紫色 | `#7C3AED` | 创新、研发、科技属性     |
| 灰蓝 | `#64748B` | 辅助文字、弱化信息      |

### 4.3 背景色

| 颜色   | 色值        | 用途            |
| ---- | --------- | ------------- |
| 页面背景 | `#F3F6FA` | App 全局背景      |
| 页面背景二 | `#EEF3F8` | 纵向渐变背景      |
| 卡片背景 | `#FFFFFF` | KPI、图表、表格卡片   |
| 浅蓝背景 | `#EAF3FF` | 提示区、弱高亮 |
| 结构边框  | `#C9D6E6` | 卡片边框、结构线    |
| 分割线  | `#E2E8F0` | 标题区、表格分割线    |
| 弱背景  | `#F8FAFD` | 表格斑马纹、辅助区     |

### 4.4 使用规则

1. 本所、本市场、当前选中对象统一使用 BSE 蓝。
2. 对标市场使用低饱和灰蓝、浅蓝或中性色。
3. 正向变化统一使用绿色。
4. 中度风险使用橙色。
5. 高风险使用 BSE 红。
6. BSE 红只用于风险、异常、负向变化和少量品牌点缀，不得作为普通大面积强调色。
7. 同一页面中主色不要超过 3 类。
8. 图表颜色应优先服务于分组和判断，不做装饰性配色。

## 5. 字体系统

### 5.1 字体选择

优先使用系统无衬线字体：

```css
font-family: -apple-system, BlinkMacSystemFont, "Segoe UI", "Microsoft YaHei", "PingFang SC", "Helvetica Neue", Arial, sans-serif;
```

### 5.2 字体层级

| 层级     | 用途    | 建议                 |
| ------ | ----- | ------------------ |
| H1     | 页面标题  | 32-40px，700，深蓝     |
| H2     | 区块标题  | 18-22px，650，深蓝     |
| H3     | 卡片标题  | 15-17px，600，深蓝     |
| KPI 数字 | 核心指标  | 28-36px，700，深蓝或科技蓝 |
| KPI 单位 | 指标单位  | 12-14px，500，灰蓝     |
| 正文     | 洞察说明  | 14-16px，400-500，深灰 |
| 注释     | 口径、来源 | 11-12px，400，灰蓝     |

### 5.3 文案规则

1. 页面标题要短。
2. 主图标题要说明分析目的，而不是只写指标名称。
3. 每个页面必须有一句话核心判断。
4. 洞察说明控制在 3-5 条。
5. 避免大段文字。
6. 指标名称保持一致，不能同一指标多种叫法。

## 6. 间距和布局系统

### 6.1 基础间距

采用 8px 间距体系：

| Token      | 值    |
| ---------- | ---- |
| `space-1`  | 4px  |
| `space-2`  | 8px  |
| `space-3`  | 12px |
| `space-4`  | 16px |
| `space-5`  | 20px |
| `space-6`  | 24px |
| `space-8`  | 32px |
| `space-10` | 40px |

### 6.2 页面留白

建议：

* 页面外边距：24px
* 卡片间距：16px 或 20px
* 卡片内边距：16px 至 24px
* 主图区应有更大留白，避免拥挤
* 明细表区可以信息密度略高

### 6.3 响应式原则

第一版优先保证桌面端展示效果。

推荐适配宽度：

* 主要设计宽度：1440px 及以上
* 最小可接受宽度：1280px
* 对外展示建议：16:9 屏幕

## 6.4 顶部导航规范

顶部导航应采用白色或近白色商务导航，并使用北京证券交易所官方 LOGO 作为左侧品牌识别。

推荐方向：

```css
height: 64px;
background: #FFFFFF;
border-bottom: 1px solid #D6E0EC;
box-shadow: 0 4px 14px rgba(15, 23, 42, 0.06);
```

规则：

1. 导航栏高度控制在 64px 左右。
2. 导航栏底部使用 `#D6E0EC` 分割线和极轻阴影。
3. 左侧使用 `www/logo_bse.png` 官方 LOGO 图片，不再使用 BSE 蓝色胶囊标识。
4. LOGO 高度建议 36-42px，宽度自适应，`object-fit: contain`，不得拉伸、变形或裁切。
5. LOGO 区与导航菜单之间保持 40-56px 清晰间距。
6. 导航菜单文字使用 `#475569` 或 `#334155`，字号约 14px，字重 600。
7. 当前页面激活态不使用圆润气泡、透明气泡、半透明背景或毛玻璃背景。
8. 激活态使用深蓝或近黑色加粗文字，并配合底部 2-3px BSE 蓝色强调线。
9. hover 状态只加深文字或使用 1px 浅蓝底部线，不出现大面积背景块。
10. 导航栏层次感来自 LOGO、字体、分割线和底部阴影，而不是厚重色块。

## 7. 卡片系统

### 7.1 卡片类型

本项目统一使用以下卡片类型：

| 卡片类型         | 用途         |
| ------------ | ---------- |
| Hero Card    | 页面一句话核心判断  |
| KPI Card     | 顶部核心指标     |
| Chart Card   | 主图容器       |
| Insight Card | 关键洞察       |
| Detail Card  | 明细表和钻取     |
| Status Card  | 风险、项目、处置进展 |
| Filter Card  | 筛选条件       |

### 7.2 卡片样式

统一规范：

```css
border-radius: 12px;
background: #FFFFFF;
border: 1px solid #C9D6E6;
box-shadow: 0 8px 22px rgba(15, 23, 42, 0.07);
```

卡片应具备 Carbon Edge 的结构感：标题区与内容区之间使用 1px 分割线；主图卡片阴影略强于普通卡片；圆角控制在 10-12px；避免毛玻璃、过度渐变和过软的圆角。

### 7.3 Hero Card

用途：

* 每个页面顶部核心判断
* 用一句话说明当前页面要表达什么

视觉：

* 浅蓝背景
* 左侧图标
* 深蓝文字
* 关键词使用科技蓝强调

示例：

> 本所是以中小市值、先进制造与创新型企业为主体的成长型市场。

### 7.4 KPI Card

KPI Card 包含：

* 图标
* 指标名称
* 指标值
* 单位
* 同比/环比变化
* 变化方向

规则：

1. 每页 KPI 数量控制在 5-6 个。
2. KPI 数字应有明显层级。
3. 同比改善使用绿色。
4. 风险上升使用橙色或红色。
5. 不要在 KPI 卡片内放过多解释文字。
6. KPI 卡片顶部或左侧使用统一 3px 品牌蓝/青色结构线。
7. 变化值可使用小型状态 badge，正向使用绿色，风险上升使用橙色或红色。

### 7.5 Chart Card

Chart Card 包含：

* 图表标题
* 口径提示
* 主图
* 图例
* 必要标注

规则：

1. 每张主图都必须能回答一个明确问题。
2. 主图优先使用大卡片。
3. 不要在一个 Chart Card 中堆叠过多图形。
4. 坐标轴、图例、颜色需要统一。
5. Chart Card 必须有清晰标题区和内容区分割线。
6. 未接入真实图表时使用正式 skeleton 状态，不使用明显虚线占位框，不以“占位”作为主标题。

### 7.6 Insight Card

Insight Card 用于放 3-5 条关键发现。

每条洞察包含：

* 编号
* 小图标
* 洞察标题
* 一句话解释

规则：

1. 洞察要解释主图，而不是重复 KPI。
2. 洞察文字应可直接用于对外汇报。
3. 洞察不超过 5 条。
4. 编号使用深蓝小方块或小矩形，不使用大圆形胶囊。
5. 洞察项之间使用细分割线或浅灰蓝底块，文案应像对外汇报材料。

## 8. 图表系统

### 8.1 图表总体规则

1. 图表要服务于页面核心问题。
2. 每页最多 1-2 张主图。
3. 辅助图只用于解释，不抢主图视觉。
4. 图例统一放在图表上方或右上角。
5. 坐标轴使用浅灰色。
6. 网格线弱化，只保留必要辅助线。
7. 本所数据必须高亮。
8. 对标对象必须弱化。
9. 风险图表使用橙红体系。
10. 图表中不要使用过多颜色。

### 8.2 气泡图规范

适用页面：

* 市场定位画像
* 市场质量画像

规则：

* 本所气泡使用科技蓝。
* 本所气泡可使用描边、光晕或标签强调。
* 其他市场使用低饱和颜色。
* 气泡大小必须有明确含义。
* 横轴和纵轴必须清楚说明。
* 图中应有 1-2 个关键解释标注。

### 8.3 四象限图规范

适用页面：

* 上市公司画像
* 市场质量画像

规则：

* 横纵轴必须代表核心判断维度。
* 四个象限必须命名。
* 象限名称应使用业务语言，而不是技术语言。
* 重点公司可标注。
* 颜色用于行业、风险或类型，不可混乱。

### 8.4 热力图规范

适用页面：

* 上市公司画像
* 市场质量画像

规则：

* 低值浅色，高值深色。
* 风险热力图使用浅橙到红。
* 经营贡献热力图使用浅蓝到深蓝。
* 热力图应配合图例。
* 不要使用过饱和色块。

### 8.5 趋势图规范

适用页面：

* 市场发展画像

规则：

* 趋势线应突出变化方向。
* 关键事件使用标记点。
* 同一图中折线不超过 3 条。
* 可用柱状图 + 折线组合表达数量和金额。
* 时间轴和趋势线要形成清晰叙事。

### 8.6 表格规范

表格适用于明细钻取，不作为页面主视觉。

规则：

* 表头使用浅蓝或深蓝。
* 行高保持舒适。
* 使用轻量斑马纹。
* 重要数值可使用小型条形图或标签。
* 风险等级使用统一标签。
* 表格默认显示重点列，更多字段可后续钻取。

## 9. 页面布局规范

### 9.1 首页总览

页面定位：

> 总览全局，不展开复杂分析。

布局类型：

* 卡片拼贴型
* 总览驾驶舱型

建议结构：

1. 顶部核心指标区
2. 四个画像入口卡片
3. 本期重点变化
4. 风险提示
5. 快速跳转

视觉重点：

* 简洁
* 开场感
* 概览性
* 不放复杂主图

---

### 9.2 市场定位画像

核心问题：

> 本交易所在资本市场中处于什么位置？

布局类型：

> 对标分析型。

页面气质：

* 横向比较
* 定位清晰
* 强调本所在哪里

主图：

1. 多市场定位气泡图
2. 本所行业市值结构图

推荐结构：

* 顶部：一句话定位判断
* 上方：KPI 卡片
* 中部左侧：多市场定位气泡图
* 中部右侧：行业市值结构图
* 下方：关键洞察
* 底部：行业/公司明细

视觉重点：

* 气泡图是第一视觉中心。
* 本所必须高亮。
* 行业结构图作为定位解释。
* 不要让表格抢主图注意力。

---

### 9.3 上市公司画像

核心问题：

> 本交易所上市公司是什么样的企业？

布局类型：

> 编辑叙事型。

页面气质：

* 像一页分析报告
* 结构和质量并重
* 强调公司群体特征

主图：

1. 行业结构与经营贡献矩阵
2. 成长性 × 盈利能力四象限图

推荐结构：

* 左侧：叙事锚点或画像目录
* 顶部：一句话判断 + KPI
* 中部：成长性 × 盈利能力四象限图
* 右侧：关键洞察
* 下方：行业贡献矩阵 + 公司明细

视觉重点：

* 四象限图承载公司质量判断。
* 行业贡献矩阵解释产业结构。
* 左侧叙事栏增强讲述感。

---

### 9.4 市场发展画像

核心问题：

> 本交易所的业务功能是否在持续增强？

布局类型：

> 时间轴推进型。

页面气质：

* 强调发展过程
* 强调阶段变化
* 强调成长动能

主图：

1. 上市—融资发展趋势图
2. 交易活跃与市场生态趋势图

推荐结构：

* 顶部：一句话判断 + KPI
* 顶部或中部：年度时间轴
* 左侧主图：上市—融资趋势
* 右侧主图：交易活跃与生态趋势
* 下方：上市审核漏斗、融资结构、区域服务、产品生态
* 底部：发展事项明细

视觉重点：

* 用时间轴讲发展。
* 用趋势线和事件标记讲阶段变化。
* 不要使用过多静态结构图。

---

### 9.5 市场质量画像

核心问题：

> 本交易所市场运行是否健康，风险是否可控？

布局类型：

> 风险驾驶舱型。

页面气质：

* 监测
* 预警
* 诊断
* 追踪

主图：

1. 市场质量状态矩阵
2. 风险类型—行业热力图

推荐结构：

* 顶部：风险状态总判断
* 上方：风险 KPI 卡片
* 中部左侧：市场质量状态矩阵
* 中部右侧：风险类型—行业热力图
* 下方：风险变化、风险叠加公司、监管处置、退市跟踪
* 底部：重点风险公司清单

视觉重点：

* 风险色应克制。
* 高风险用红色，中风险用橙色。
* 风险矩阵和热力图是双主图。
* 明细表重点体现“公司—风险类型—处置进展”。

## 11. 状态和风险标签

### 11.1 状态标签

可参考 Atlassian 的状态表达，但视觉上保持 Fluent 风格。

常用状态：

| 状态  | 颜色  |
| --- | --- |
| 正常  | 绿色  |
| 关注  | 橙色  |
| 风险  | 红色  |
| 进行中 | 科技蓝 |
| 已完成 | 绿色  |
| 待确认 | 灰蓝  |

### 11.2 风险等级

统一风险等级：

| 风险等级 | 样式          |
| ---- | ----------- |
| 低风险  | 浅绿背景 + 绿色文字 |
| 中风险  | 浅橙背景 + 橙色文字 |
| 高风险  | 浅红背景 + 红色文字 |

规则：

1. 风险等级不使用过多颜色。
2. 同一风险等级在全项目中样式一致。
3. 风险标签应简短，不写长句。
4. 风险解释放在洞察区或明细表中。

## 13. 禁止事项

1. 禁止做成传统 Shiny 灰色控件风格。
2. 禁止所有页面使用完全相同布局。
3. 禁止每页堆叠过多图表。
4. 禁止过度使用红色。
5. 禁止过多渐变、发光、粒子背景。
6. 禁止在 R 文件中大量写内联 CSS。
7. 禁止主图和辅助图视觉权重不分。
8. 禁止表格占据主视觉。
9. 禁止指标名称前后不一致。
10. 禁止为了装饰牺牲可读性。
11. 禁止导航栏使用圆润 pill、透明气泡或半透明块作为当前页面激活态。
12. 禁止使用过度柔和、过度圆润的通用 SaaS 风格。

## 14. Codex 执行要求

当 Codex 开发或修改 UI 时，应遵循：

1. 开始 UI 修改前先阅读本文件。
2. 页面视觉以 Microsoft Fluent 为主。
3. 结构线和图表规范参考 IBM Carbon Edge。
4. 页面组织参考 Ant Design Pro。
5. 状态和流程组件可少量参考 Atlassian。
6. 所有新增样式优先写入 `www/custom.css`。
7. 所有通用 UI 组件优先写入 `R/ui_components.R`。
8. 不要破坏既有模块化结构。
9. 不要为了美观增加过多无意义元素。
10. 修改完成后检查 app 是否可以运行。
11. 进行 UI 修改前必须阅读 UI_STYLE_GUIDE.md，并遵守其中的 Fluent Light + Carbon Edge + BSE Brand 视觉规范。
